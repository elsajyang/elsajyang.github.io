a developer's nightmare

I wanted to build a website. This website. And somehow I ended up googling "how to run a program in terminal."

... while trying to run the command browser-sync, a Node package I installed locally to help refresh pages automatically for web development, on the command line. The answer could not be found on browser-sync's website, but I quickly found BrowserSync: command not found after installing locally.

The answer I found to my question worked, but I couldn't say I fully understood it because I never fully understood PATH. I tried to comprehend PATH again, but this time in a roundabout way—I googled "how to run a program in terminal."

Fortunately, I was able to put together the pieces after reading why do we use a "./" (dot slash) to execute a file in Linux/UNIX and rereading What are PATH and other environment variables, and how can I set or use them? And luckily I found an answer that was beautifully put. But like most answers I find, the answer is only as good as the question posed. And the question posed on the StackExchange post assumed that I, the reader, had to have some exposure to or a general idea of the following:

  • PATH, the environment variable, and how an OS (operating system) uses it
  • absolute paths vs relative paths
  • path resolution
  • how one runs commands via command line
  • how programs are "usually" run

Can you see how the first bullet point is tragic? (And yes, I learned PATH by relearning PATH) Can you see how that last bullet point is ironic? Imagine if I were a novice who simply wanted to know "how to run a program in terminal?"

I'm reminded of "CS162: Operating Systems," a course I took as an undergraduate. I can still remember how utterly confused I was trying to implement Project 1: User Programs and System Calls and HW2: Shell. The confusion was conceptual, and some of it plain jargon—what is a shell? what are system calls? In the end, it was the select resources I perused and my fellow honor society peers I seeked out for help that made things click.

They are always saying "just Google it." They aren't wrong—I manage to find what I'm looking for. Problem solving is fantastic. Struggling while problem solving is necessary. But mindlessly trying to solve an undefined problem is not.

Often the problem is, where does one start the search? What jargon or keywords should we use in the search process? The search process often requires my undergraduate CS knowledge and experience, or the intuition gained from exposure, or copious amounts of time scanning through each link until I finally get a sense of what's happening.

I spent hours trying to configure and run browser-sync only to find that the solution was quite simple. The solution isn't well documented, and I contend requires both a developer's intuition for command line and conceptual knowledge of how to run programs.

One experiences both beauty and torment as a developer, as one does riding a rollercoaster. When things don't work, it's frustrating. But when they do, it's pure bliss. Then you find yourself back where you started, like code in a while (true) loop.

I've arrived back where I started. But this time it's different—I'd ridden the ride.

The following posts hope to consolidate my findings in one place for my reference as well as streamline the developer's ride for someone else.
