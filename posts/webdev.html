A guide to building a static website from scratch.

To understand how to build a website, it is useful to understand the infrastructure which supports it. We start with the Web.

The Web and the Internet are often used interchangeably, though they are actually two different things.

What do you associate with the Web? The Internet? What would you say differentiates them?

The short answer is, the Internet is a network of computers that uses the Internet Protocol to communicate. The Web on the other hand is a service that we access over the Internet. The Web is defined by how it organizes and accesses information. Information on the web, known as web resources and documents, are identified by URLs, Uniform Resource Locators and may interlink (linked meaning connected or lead to one another), hence the name web.

The Internet, Internet Protocol

The Internet is a network of computers that uses the Internet Protocol to communicate.

A group of computers that are connected to one another and communicate using a common protocol is called a computer network. These computers are physically wired, or receive and transmit wireless radio-frequencies, or use other technologies for telecommunication.

Today the Internet is a global network of computer networks. On the hardware side, it is made up of computers and special computers, like routers and modems, serviced by Internet service providers who communicate data over cables of length in the thousands of kilometers. On the software side, the Internet Protocol (IP for short) defines how computers communicate data with one another, that is, how to route Internet traffic. A computer is addressed using a fixed-length number, called an IP address, which is assigned in some way by the network. The Internet routes data, which are called packets, using the IP addresses of the source and destination computers.

The term computer refers to machines, so it includes anything with a microprocessor (CPU), like smartphones, smart devices, game consoles, destop and personal computers, and servers.

How are IP addresses assigned? Can a device's IP address change over time?

The Web

The World Wide Web, according to Mozilla, is a service that exists atop the Internet and which we access through the Internet. The Web is defined by how it organizes and accesses information. Information on the web, known as web resources and documents, are identified by URLs, Uniform Resource Locators and may interlink (linked meaning connected or lead to one another), hence forming a web.

Along with the Web, the Internet has allowed services that permeate our lives, including email, social media, e-commerce, digital music and news, video streaming, and more.

Websites, Web Browsers, Web Servers

When we wish to visit a website like MDN Web Docs, we type in a URL, a Uniform Resource Locator, into a web browser. The web browser does some magic which, upon completion, displays the website onto our screen. The website is made up of web pages—the page we are served when we visit https://developer.mozilla.org/en-US/ is one such page, often called the landing page or the homepage. The page links to other pages.

A web page is actually a document, called an HTML document. It lies on a machine somewhere called a web server, and your browser uses the url to identify the machine. Machines that store and deliver web pages are called web servers. That is, a web server hosts websites. "Hosting" means that all the web pages and their supporting files, including HTML documents, images, CSS, and JavaScript scripts, are available on the server. Web servers host one or more websites, and some common web servers include Apache HTTP and nginx.

Web servers communicate using HTTP (Hypertext Transfer Protocol). Web servers are often called HTTP servers for that reason. Your browser sends an HTTP request—a message with a request—to the machine at the requested url, and the web server responds back with an HTTP response.

HTTP, Hypertext Transfer Protocol, is a communication protocol. A protocol is a set of rules that define how to communicate. Like a language, it defines grammar, or syntax, and semantics, like what each string or keyword means. HTTP was designed for web browsers and web server to communicate. It defines its communicators as clients and servers, a common model, and is stateless, meaning a HTTP packet contains all the information a client or server needs to perform a task.

Web servers are machines, and like all machines one needs to manage its resources. In general, hosting applications over the Internet involves managing a virtual or physical server... as well as the operating system and other hosting processes. Usually web developers write extra code to manage web servers. An alternative to having to maintain servers of any sort is offered by serverless computing. Serverless computing is cloud computing in which the cloud provider runs servers and dynamically allocates them according to the amount of computation and storage users require, who pay according to their server usage. Serverless computing relieves users from having to own or manage any server infrastructure, which was the paradigm of the past).

The takeaway is, the Internet is powerful, the Web is still quite novel, and the possibilities to innovate are vast. Knowing how the Web and the Internet works is important to understanding how to build proper, secure, and fast websites.

My first idea for a website was:

Create an interactive webapp that looks and behaves like a terminal. The website can take inputs from the keyboard or mouse and display the output. Users can login as guests or create an account. They can execute commands to run programs or projects I've created.

As I searched "how to create a terminal like website," I soon learned the difference between a static and dynamic website. A static website is one where all the content is stored at a server and users who visit a website simply get served those exact files. However, parts of my website were features of a dynamic website. For instance, in order for users to create accounts, I needed to store that information somewhere, meaning I needed a database from which to store and fetch data. I also needed some way for my website to run a program I wrote, which I'd have to store on a server, and display that output.

Soon, I scratched the interactive web app idea when I realized I can't really host a free dynamic serverless website. There was also more to learn than I thought, so I kept things simple. I decided I would host a static website inspired by terminal UX/UI on GitHub Pages, because it is free and hassle-free.

My objective was to learn web development, so I chose to start by building a website from scratch in HTML and CSS.

Although PHP might be more suitable for personal blogs, one might choose Node.js for simplicity and consistency. That is, since Javascript is both a front end language (Javascript) and a server language (Node.js), one can use just one language throughout your stack, which would be called an end-to-end Javascript stack.

3: HTML

Keywords

  • HTML content organization
  • section vs article vs div
  • link vs a

It is simpler to begin with just HTML and ensure the content is organized and tagged properly. HTML should read well even if it doesn't look nice yet. The comprehensive guides w3schools's HTML tutorial and Mozilla Web Docs: HTML are perfect places to start.

Some of the things I picked up and incorporated include:

  • All text should be enclosed in a tag.
  • Use tags where appropriate, like header, footer, section, article for content, nav for navigation, andd even address for contact information
  • Headers and footers are not just for the site, but also for articles
  • Define a main tag to include the main content. It might help to add a link to jump directly to the content to aid screen readers
  • There's a difference between section and article. Content in article should be independent, self-contained, and reusable
  • div is best reserved for styling, though nothing and no one says so strictly
  • The difference between <link /> and <a></a>
    • <link /> is an empty element, hence the difference in tag syntax. Used in the <head>, it defines the relationship between the current document and an external resource.
    • <a></a> is an HTML element that has an end tag. It is used in the body and creates an object on the page.
  • A way to debug links in Chrome is to look at the bottom left hand corner where a black box with the link appears.

I found it helpful to look at other websites to see what conventions they follow, if any, such as how they labeled section vs article, or how they organized their main content.

Natural Property Value vs Browser Default

  • Every CSS property has a natural value. Some properties like font-family, position, color naturally inherit values, meaning they set the value applied to the selected element to be the same as that of the element's parent.
    • body { font-family: san-serif; } Every element in the <body> tag will inherit font-family.
  • Other properties like display, position do not inherit from their parents. They do still have a natural value, which is its initial value. The initial value is often defined by the browser default.
    • For instance, display: inline; position: static; are these properties' natural values.
  • We can change a CSS property's natural value. Every CSS property accepts four special universal property values to control inheritance: inherit, initial, unset, (and revert which has limited browser support).
  • Type selectors, known as element selectors, select an HTML tag/element. Element selectors are predefined with certain property values in the user agent stylesheet, which is the browser default's CSS rules.
    • By default, <div> is applied display: block
  • When we define a selector using the CSS syntax selector { declaration block }, it is important to note that it isn't the entire rules applied to the selectors which are overwritten, but rather only the properties which are the same.
  • We follow the principles cascade, specificity, and inheritance on the browser default.

Fonts

There are many ways to import a font.

Crumbs: One can use an open source font on the web (the ones I chose are from Google Fonts, Assistant and Fira Mono) and import those links in your .css stylesheet, or one can include font files with your website content and import those files using @font-face.

It is important to exercise good font size setting semantics. In css, there are two commonly used units, px for pixels and em. px is a static measure, meaning the text displayed will not change its size when a window or screen is resized. But what if a user wishes to zoom a page and expects all contents to enlargen? We'd choose to use em, a unit that is relative to, or a percentage of, some reference point.

For more on how to set em font sizes, read Font sizing with rem and How to Size Text in CSS.

5: Deploying on GitHub Pages

Once all your files are in order, we are ready to deploy them to GitHub.

To deploy a website on GitHub, follow the instructions found on GitHub Pages. The gist is:

  • Create a GitHub repository on github.com in the form of <username>.github.io, or one that follows the guidelines listed on GitHub Pages.
  • Once the remote repository has been created, clone the repo to your local machine.
  • Add, commit, and push your local changes to the remote repo

It is good practice to save changes to your website as you develop each new section.
